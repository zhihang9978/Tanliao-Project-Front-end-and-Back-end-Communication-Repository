## 2. P0-10 Cookie HttpOnly + Secure + SameSite ✅

### 改动(运维层,nginx)
`/etc/nginx/sites-available/tantan.conf` 6 个 server 块全部加:
```nginx
proxy_cookie_flags ~ httponly secure samesite=lax;
```
nginx 1.24.0 原生支持(1.19.3+)。所有从后端来的 `Set-Cookie` 自动加 flags。

### 客户端影响
- **浏览器 / Web tioim**:
  - JS `document.cookie` 读不到 sessionId(防 XSS 偷)
  - 仅 https 才发(已是默认)
  - 跨站 POST 不带 cookie(防 CSRF)
- **Flutter App / 原生客户端**:
  - 一般用 token 认证(在 header 里),不依赖 cookie → **无影响**
- **如发现 Web 端登录态丢失** → 排查 SameSite=Lax 是否过严(可改 None,但需 Secure)

### 验证
- nginx 1.24 支持 `proxy_cookie_flags` 验证
- `nginx -t` 配置语法 OK
- `nginx -s reload` 无回归
- 真实功能验证待 Codex 用浏览器实测(devtools → Application → Cookies 看 flags)

## 3. P0-11 WebSocket Origin 白名单 ✅

### 改动
[ImWsMsgHandler.java](服务端) `handshake()` 第一步加 Origin 白名单:
```java
String origin = request.getHeader("Origin");
if (StrUtil.isNotBlank(origin)) {
    String allowed = P.get("ws.allowed.origins", "");
    if (StrUtil.isNotBlank(allowed)) {
        // 逐项匹配 anjuke.site / api / web / admin / res 子域
        if (!匹配) {
            log.warn("WS-ORIGIN-DENY ...");
            Tio.remove(channelContext, "Origin not allowed");
            return null;
        }
    }
}
```

### 当前 anjuke.site 白名单
```
ws.allowed.origins=https://anjuke.site,https://api.anjuke.site,https://web.anjuke.site,https://admin.anjuke.site,https://res.anjuke.site
```

### 客户端影响
- **Flutter App 客户端**:
  - 原生 WebSocket 不发 Origin header(浏览器才有)
  - `if (StrUtil.isNotBlank(origin))` → 跳过校验 → **无影响**
- **Web tioim / tioim-small / mg-page**(浏览器内 WebSocket):
  - 浏览器自动带 `Origin: https://anjuke.site` 等 → 在白名单内 → 通过
  - 跨站攻击 origin → 拒绝
- **如 Codex 之后启用其他子域** → 通知后端把 origin 加入白名单
- 如 Codex 本地开发 / staging 环境域名不同,后端可单独配 ws.allowed.origins(开发环境留空 = 不校验)

### 验证
- 配置已写入 anjuke 运行时 properties
- 9326 当前 2 个 ESTABLISHED 连接持续 = 重启后客户端立即重连成功 = 校验对真实客户端无误伤
- 日志关键字:`WS-ORIGIN-DENY`(若有触发说明拦截了攻击 origin)

## 4. P1-14 access-url-role 默认 deny 配置开关 ✅

### 改动
[AccessCtrlService.java](服务端) 末端 `else { return true; }` 改:
```java
} else {
    String defaultPolicy = P.get("access.default.policy", "allow");
    if ("deny".equalsIgnoreCase(defaultPolicy)) {
        log.warn("ACCESS-DEFAULT-DENY: 路径未在 access-url-role 配置,默认拒绝 ...");
        return false;
    }
    return true;
}
```

### 当前状态
**anjuke.site 仍是 allow(向后兼容)**。原因:
- bs-server 100+ endpoint,access-url-role.properties 仅 ~10 条规则
- 直接切 deny 会让大量未配置规则的合法 endpoint 全 403
- 需先全 endpoint 对账(预计 1 天工作)

后续步骤(后端单独跑):
1. grep 全 `@RequestPath` 列 endpoint 清单
2. 对照 access-url-role.properties 补缺失规则
3. anjuke runtime 加 `access.default.policy=deny`
4. 监控 24h 看 `ACCESS-DEFAULT-DENY` 日志发现遗漏

### 客户端影响
- **当前**:无(默认 allow,行为完全等同今天)
- **后端切 deny 后**(未来):若访问的 endpoint 未配规则,Codex 客户端会收到 401/403
- 后端切前会先 grep 全 endpoint 对账,Codex 不需主动配合
- 如 Codex 之后新增依赖某个 endpoint 而后端忘加规则 → 用本仓库提 issue
